Reads whatever planning material exists (pptx/pdf/md) — or runs a bounded schema interview when none exists — and produces the three files an unattended loop needs — CLAUDE.md (what to build it with), requirements (what to build), PROMPT.md (how the loop runs) — then isolates them in a git worktree and prints the exact ralph-loop command. Announces its question plan up front, then stops the user at most twice: slice selection when the planning doc is large, and one combined approval of requirements + CLAUDE.md. Everything else is automatic. Never starts the loop itself.

# Ralph Handoff — 요구사항까지만 승인하고 나머지를 맡긴다

**당신은 "뭘 만들지"까지만 정합니다. 나머지는 자고 일어나면 돼 있습니다.**

```
당신 │ (기획서 크면) 조각 선택 · (없으면) 인터뷰 3~5문 → 요구사항+CLAUDE.md 승인   ← 멈춤 최대 2번
     ▼
랄프 │ 개발방향 → 구현계획서 → 코드 → 검증                   ← 무인
```

<HARD-GATE>
**멈춤(승인 게이트)을 두 번 넘게 만들지 마세요.** 조각 선택(기획서가 클 때만)과 최종 승인, 그게 끝입니다. **질문은 Step 0 에서 예고한 수 안에서만** 하세요 — 기획서가 없으면 예산 인터뷰(3~5문, Step 2)가 그 예고 안에 들어갑니다. 사용자는 "알아서 빨리"를 원해서 이걸 씁니다.

**요구사항과 `CLAUDE.md` 는 반드시 승인받으세요.** 루프가 몇 시간 도는데 방향이 틀리면 전부 버립니다. 단 **둘을 묶어서 한 번에** 물으세요.

**현재 브랜치를 절대 건드리지 마세요.** 삭제는 워크트리 안에서만. 원본이 남아야 되돌릴 수 있습니다.

**루프를 직접 시작하지 마세요.** 마지막에 명령을 **출력만** 합니다. 루프는 사람이 겁니다 — `ralph-loop` 가 그 명령을 `hide-from-slash-command-tool` 로 막아둔 이유이기도 합니다.
</HARD-GATE>

### 1. 여기가 프로젝트 폴더여야 합니다

**이 스킬은 `cwd` 를 봅니다.** 자기 플러그인 폴더가 아니라요.

```bash
git rev-parse --git-dir >/dev/null 2>&1 || echo "git 저장소가 아님"
```

git 저장소가 아니면 — **`git init` 을 대신 하지 말고** 아래를 안내하고 중단하세요. 어디에 뭘 만들지는 사용자가 정할 일입니다.

```bash
mkdir -p <프로젝트>/docs && cd <프로젝트>
git init
mv <기획서>.pptx docs/                       # 있으면
printf 'docs/*.pptx\ndocs/*.pdf\n' > .gitignore
git add .gitignore && git commit -m "chore: 초기"
```

> **기획서는 프로젝트 안에 있어야 합니다.** 사용자가 플러그인 폴더에 기획서를 두는 실수가 실제로 있었습니다 — 설치된 플러그인은 `~/.claude/plugins/cache/` 로 복사되는 거라 소스 폴더의 문서는 아무도 못 읽습니다. **이 스킬은 `cwd` 기준으로만 파일을 찾으세요.**

## ★ Step 0 — 시작하자마자 질문 계획을 예고하세요

**사용자가 제일 싫어하는 건 질문 수가 아니라 "언제 끝나는지 모르는 것"입니다.** 첫 응답에 이 한 덩어리를 먼저 내세요. 그래야 사용자가 대응할 준비를 합니다.

```
ℹ️ 여쭐 건 <N>번입니다. 나머지는 알아서 하겠습니다.

   ① 어느 조각을 넘길지    ← 기획서가 <N>장이라 통째로는 PRD 가 안 됩니다
   ② 요구사항 + CLAUDE.md 승인   ← 루프가 몇 시간 돌아서, 여기가 마지막 방어선

   자동으로 할 것: 기획서 읽기 · 스택 추출 · 요구사항 작성 · 커밋 ·
                  워크트리 · PROMPT.md · 실행 명령 출력
```

**기획서가 작아서 ①이 필요 없으면 "1번입니다" 라고 하세요.** 실제 개수를 세서 말하세요 — 지어내지 말고.

**기획서가 아예 없으면** → ① 대신 **스키마 인테이크**입니다. 예고를 이렇게: *"기획 자료가 없어 빈 슬롯 N개를 최대 3~5문으로 채우고, 나머지는 기본값으로 제안 → 그다음 요구사항+CLAUDE.md 승인 1번."* (규칙: [`references/requirements-schema.md`](references/requirements-schema.md))

이 예고 없이 질문부터 던지면 **사용자는 "이게 언제 끝나지" 하며 지칩니다.**

## ★★ 사용자를 딱 두 번만 멈춰라 ★★

**이 스킬의 성패는 멈춤 수입니다.** 사용자는 "알아서 빨리"를 원해서 무인 루프를 씁니다. 물을 게 많으면 그냥 자기가 하고 말죠.

| 멈추는 곳 | 왜 못 빼나 |
|---|---|
| **① 조각 선택** *(기획서가 클 때만)* | 30장짜리 발표자료를 어떻게 자를지는 **사람만** 압니다. 통째로는 PRD 가 안 돼요 |
| **② 요구사항 + `CLAUDE.md` 묶어서 1번** | 루프가 **몇 시간** 돕니다. 방향이 틀리면 **전부 버립니다.** 이게 마지막 방어선 |

**그 외는 전부 알아서 하세요. 묻지 마세요:**

| 묻지 말 것 | 대신 |
|---|---|
| "어디까지 맡길까요?" | **기본 = 요구사항만 주고 나머지 전부.** 사용자가 "코드만" 이라 말했을 때만 다르게 |
| "기획 자료 있어요?" | **`docs/` 를 그냥 찾아보세요.** 있으면 읽고, **없으면 스키마 인테이크**(Step 2 · 예고된 3~5문). 한 줄로 때우지 마세요 |
| "PRD? 소크라테스?" · "카테고리?" · "질문 계획 볼까요?" | **`brainstorming` 에 위임하지 마세요.** 게이트가 5개 더 생깁니다. 아래 형식으로 **직접 쓰세요** |
| "워크트리 만들까요?" | **묻지 말고 판단하세요.** 지킬 코드가 있으면 만들고, 빈 프로젝트면 브랜치만 (Step 5) |
| "커밋할까요?" | 그냥 하세요. 워크트리 안입니다 |

> **`brainstorming` 위임은 사용자가 명시로 요청할 때만.** 그 스킬은 라우터·모드·카테고리·질문계획·승인으로 게이트가 5개입니다. 여기선 **직접 쓰는 게 맞습니다** — 어차피 사용자가 마지막 승인에서 다 봅니다.

## ★ 무인 루프가 받아야 하는 건 3개다

**루프는 채팅을 못 봅니다. 파일에 없으면 없는 겁니다.** 이 스킬의 존재 이유가 이 표입니다.

| 파일 | 답하는 질문 | 없으면 |
|---|---|---|
| **`CLAUDE.md`** | **뭘로** 만드나 (스택 · 구조 · 원칙) | **루프가 스택을 지어낸다** |
| **`<slug>-requirements.md`** | **뭘** 만드나 (기술 중립) | 만들 게 없다 |
| **`PROMPT.md`** | 루프가 **어떻게 도나** | 무한 루프 / 거짓 완료 |

> **실제 사고** — `CLAUDE.md` 없이 넘길 뻔했습니다. 요구사항은 **일부러 기술 중립**이라 (`intent-locked-workflow` 의 `brainstorming` 규칙: *"기술 결정은 여기 박지 마세요"*) 스택 키워드가 **0건**이었어요. 스택이 **채팅에만** 있었던 겁니다. 그대로 돌렸으면 Flask 든 Express 든 루프가 자기 맘대로 골랐습니다.
>
> **요구사항이 기술 중립인 건 버그가 아니라 규칙입니다.** 스택은 원래 `CLAUDE.md` 자리예요.

## Step 2 — 기획 자료 (묻지 말고 찾으세요)

```bash
ls docs/*.pptx docs/*.pdf docs/*.md 2>/dev/null
```

**있으면 그냥 읽으세요** (아래 pptx 추출). **없으면 → 스키마 인테이크** — 아래 ★ 절에서 빈 슬롯을 예산 인터뷰로 채웁니다 (한 줄로 지어내지 마세요). `.pptx` 는 zip 이라 파이썬으로 텍스트를 뽑을 수 있습니다:

```bash
python3 -c "
import zipfile, re, sys, glob
z = zipfile.ZipFile(glob.glob(sys.argv[1])[0])
slides = sorted([n for n in z.namelist() if re.match(r'ppt/slides/slide\d+\.xml$', n)],
                key=lambda x: int(re.search(r'\d+', x.split('/')[-1]).group()))
for i, s in enumerate(slides, 1):
    t = re.sub(r'<[^>]+>', ' ', z.read(s).decode('utf-8', 'ignore'))
    print(f'--- slide {i}\n{re.sub(chr(92)+chr(115)+chr(43), chr(32), t).strip()}')
" '<경로>'
```

### ★ 기획서가 없으면 — 스키마 인테이크 (자유 문답이 아니다)

기획서가 없으면 정보가 당신한테서 와야 합니다. 하지만 **흥미 위주로 무한정 묻지 마세요.** [`references/requirements-schema.md`](references/requirements-schema.md) 의 **빈 슬롯을 채우는 최소 질문**만 합니다.

no-doc 엔 조각 선택(멈춤 ①)이 없습니다 — **인터뷰 → 요구사항 → 승인(멈춤 ②) 한 번**으로 끝나고, 질문은 Step 0 예고 안에서만 합니다 (HARD-GATE).

1. **먼저 예고** (Step 0 형식): "빈 슬롯 N개, 최대 3~5문, 나머지는 기본값+근거."
2. **질문 1개 = 슬롯 1개 이상.** persona → 핵심 저니 → 만들거나-부수는 제약 하나. **가장 추측을 줄이는 것부터.** 못 닫는 흥미 질문 금지.
3. **예산 소진 → 빈 슬롯을 "기본값 + 근거 1줄"** 로 채우고 `[기본값]` 표시. 승인 게이트에서 한 번에.
4. **그린필드로 스택을 모르면** 스택도 같은 예산 안에서 1문 물어 `CLAUDE.md` 로 보냅니다 — Step 4 가 **따로 멈춰 묻지 않게** (요구사항은 기술 중립, 스택은 CLAUDE.md 자리).
5. **완료 산출물은 Step 3 의 요구사항 형식과 동일** — 이후 흐름(CLAUDE.md → 승인 → 커밋 → PROMPT.md)은 그대로.

### ★ 읽자마자 — 기획서가 커밋될지 **확인하세요**

권하지 말고 **검사하세요.** 기획 자료는 회사 문서인 경우가 많고, 공개 저장소에 들어가면 되돌릴 수 없습니다.

```bash
for f in docs/*.pptx docs/*.pdf docs/*.hwp docs/*.xlsx; do
  [ -e "$f" ] || continue
  git check-ignore -q "$f" || echo "⚠️ 커밋될 수 있음: $f"
done
```

**하나라도 걸리면 — 묻지 말고 `.gitignore` 에 추가하고 커밋하세요.** 안전한 쪽이 기본값입니다.

```bash
printf 'docs/*.pptx\ndocs/*.pdf\ndocs/*.hwp\ndocs/*.xlsx\n' >> .gitignore
git add .gitignore && git commit -q -m "chore: 기획서를 gitignore 로 차단"
```

그리고 사용자에게 한 줄로 알리세요 — "기획서가 커밋될 수 있어서 `.gitignore` 에 넣었습니다. 로컬엔 그대로 있습니다."

> **실제로 두 번 터졌습니다** — 회사 발표자료가 공개 저장소에 들어갈 뻔했어요. **기획서는 로컬에, 뽑아낸 요구사항만 git 에.**

**읽은 내용은 요약해서 보여주세요** (몇 장에서 뭘 봤는지). 조각 제안의 근거가 됩니다.

### ★ 조각은 **수직으로** 자르세요 (층으로 자르지 마라)

한 조각은 **기능 하나가 입력 → 로직 → 화면을 관통**해야 합니다. "엔진 층 / 화면 층" 으로 자르면 로직 조각은 **정의상 아침에 볼 게 없습니다** (아래 ★★ 절의 실제 사고).

```
❌ 층으로 (수평)              ✅ 기능으로 (수직)
   S2 = 계획 엔진 (화면 X)      S2 = "여행 계획 하나를 만들어 화면에 띄운다"
   S3 = 일정 화면                     → 엔진 + 최소 화면이 한 조각
```

### ★ 조각을 제안할 때 — 축이 **두 개**입니다

기획서가 커서 조각을 나눠 제안할 때, **두 축을 따로 매기세요.** 흔한 실수는 하나만 보는 겁니다.

| 축 | 뜻 |
|---|---|
| **무인 적합도** | 랄프가 **인프라 없이 검증**할 수 있나 (순수 함수 ★★★ / DB·모델 필요 ★☆☆) |
| **눈으로 확인** | 아침에 사람이 **뭘 볼 수 있나** (화면 ✅ / 테스트 출력만 ❌) |

**이 둘은 자주 반대입니다.** 순수 로직은 랄프가 잘 검증하지만 **볼 게 없고**, 화면은 눈에 보이지만 **랄프가 렌더를 검증 못 합니다** (canvas 는 브라우저 없이 판정 불가).

```
| # | 조각 | 무인 적합도 | 눈으로 확인 |
|---|------|------------|------------|
| 1 | 검색 파이프라인 | ★★★ 순수 함수 | ❌ pytest 출력만 |
| 6 | 채팅 프런트     | ★★☆ 백엔드 mock 필요 | ✅ 브라우저 |
```

### ★ 조각을 **둘 이상 넘길 생각이면** — 병렬 판정을 같이 하세요 (기본은 1개)

**관측 1건**(순수 로직 조각 · 45분)에서 조각 하나가 한 시간 안쪽이었습니다. 여럿을 순차로 돌리면 그만큼 쌓이니, **안 겹치는 조각은 동시에** 돌릴 수 있습니다. 판정은 **두 축**입니다:

```
| 조각 | 건드릴 파일 | 무인 적합도 | 병렬 |
|------|------------|------------|------|
| S2 엔진 | src/planner/** · tests/engine/** | ★★★ | ✅ 교집합 0 · env 불필요 |
| S3 화면 | src/screens/** · src/planner/types.ts | ★☆☆ 브라우저 | ❌ S2 와 겹치고 사람 env 필요 |
```

**파일 교집합이 0이어도, 사람이 env(DB·결제·브라우저)를 꽂아야 green 인 조각은 병렬 후보가 아닙니다** — 밤새 돌려도 BLOCKED 로만 끝나니까요.

**병렬엔 전제가 셋** — ①공유 척추(`package.json`·lockfile·스키마·테스트 디렉터리)를 **분기 전에** 세울 것 ②`REVIEW/VERIFY/BLOCKED` 를 슬라이스 폴더로 내릴 것 ③`PROMPT.md`·`CLAUDE.md` 는 **머지 대상이 아님**. 실측 근거와 절차: [`references/parallel-slices.md`](references/parallel-slices.md)

### ★★ 눈으로 볼 게 없는 조각이면 — **묻지 말고 요구사항에 넣으세요** ★★

**제안하지 마세요. 넣으세요.** "넣을까요?" 라고 물으면 멈춤이 3번이 되고, 그럼 사용자가 지칩니다. **초안에 넣어두면 승인 게이트에서 보고 빼면 됩니다** — 빼는 건 한 마디면 되지만, 없는 걸 알아채고 요청하는 건 어렵거든요.

**판정은 기계적으로:** 고른 조각의 `눈으로 확인` 이 ❌ 면 → **확인용 화면 FR 을 무조건 추가.**

> **★ "화면은 다음 조각이 담당한다" 는 면제 사유가 아닙니다 — 실제로 터졌습니다.**
>
> Wayfare 를 조각낼 때 **층으로 잘랐습니다** — S2=엔진 층, S3=화면 층. 그리고 *"화면은 S3 몫"* 이라며 S2 의 확인용 화면을 뺐습니다. 결과: **테스트 16개 green, 아침에 볼 것 0.** 사용자의 첫 반응이 *"화면 안 만들어져 있어?"* 였습니다.
>
> **조각은 층이 아니라 기능으로 자르세요(수직).** 한 조각은 *입력 → 로직 → 화면*을 관통해야 합니다. 층으로 자르면 로직 조각은 **정의상** 볼 게 없고, "다음 조각이 화면을 만든다"는 말은 **오늘 밤의 결과물이 보이지 않는다**는 뜻일 뿐입니다.
>
> 불가피하게 로직만 넘겨야 한다면(다른 조각이 그 로직을 기다리는 경우) — **그때야말로 확인용 화면이 필수**입니다. 면제가 아니라 반대입니다.

> **실제로 놓쳤습니다** — 검색 파이프라인 조각을 넘겼더니 요구사항이 *"§2 사용자 스토리: 해당 없음 — 내부 컴포넌트라 외부 사용자 노출 화면이 없음"*, *"§5 범위 밖: 어드민 / **프런트** / 배포"* 로 나왔습니다. 조각의 성격만 보면 맞는 말이라 **아무도 안 막았어요.** 결과: 22개 테스트는 통과했는데 **사람이 볼 게 아무것도 없습니다.**
>
> 요구사항이 "내부 컴포넌트"라고 스스로 규정하면 화면은 자연히 잘려나갑니다. **그래서 조각 성격에 맡기면 안 되고, 규칙으로 박아야 합니다.**

승인 게이트에서 이렇게 알리세요:

```
📄 요구사항 — <슬러그>
   …
   FR-N (확인용 화면)  ← 이 조각은 전부 함수라 볼 게 pytest 출력뿐이라 제가 넣었습니다.
                         빼시려면 말씀해주세요. 대신 아침에 볼 게 없습니다.
```

요구사항에 이렇게 넣습니다:

```markdown
- **FR-N (확인용 화면)**: 입력창에 <입력>을 넣으면 <중간 결과>와 <최종 결과>가 표시된다.
## 5. 범위 밖
- **디자인·스타일링** — 동작 확인용 화면이다. 꾸미지 않는다
## 6. 수용 기준
- **AC-N [e2e]**: 화면이 열리고, <입력>을 넣으면 <결과>가 표시된다  ← Playwright 가 클릭해 판정 (브라우저 없으면 사람이 눈으로)
```

**"디자인은 범위 밖" 을 반드시 넣으세요.** 안 그러면 루프가 몇 시간을 CSS 에 씁니다.

### ⚠️ `§5 범위 밖` 에 "프런트" 를 통째로 쓰지 마세요

**이게 실제로 화면을 죽인 문장입니다:**

```markdown
❌  - 어드민 / 프런트 / 배포            ← 확인용 화면까지 같이 잘려나감
✅  - 어드민 · 배포 · 제품용 화면       ← 확인용 화면(FR-N)은 범위 안
      (FR-N 의 확인용 화면은 예외 — 동작을 눈으로 보기 위한 최소 UI)
```

**요구사항 §2 에 "화면 없음" 이라고 쓰지도 마세요.** *"내부 컴포넌트라 외부 사용자 노출 화면이 없음"* 이라고 스스로 규정하면 **그 뒤로 화면이 자연히 잘려나갑니다.** 확인용 화면이 있으면 §2 에 그 사용자 스토리를 쓰세요 — *"개발자로서, 이 파이프라인이 진짜 도는지 브라우저로 확인하고 싶다."*

**렌더 확인은 브라우저가 있냐 없냐로 갈립니다** (Step 6-8 이 이걸 배선합니다):

- **브라우저가 있으면** — `[e2e]` AC 를 Playwright 로 **실제로 클릭해 검증**하고, `VERIFY.md` 에 출력·스크린샷 경로를 증거로 남깁니다.
- **브라우저가 없으면** — 오늘과 똑같습니다. 완료 조건에 넣지 말고 `BLOCKED.md` 의 *"사람이 확인해야 할 것"* 으로 빼되, **확인 절차(주소 · 뭘 넣고 · 뭐가 보여야 하는지)까지 적어주게** 하세요.

**둘 중 뭐가 될지는 루프가 런타임에 판정합니다** (`npx playwright` 가 도나 · dev 서버 명령이 `CLAUDE.md` 에 있나). 그래서 이 기능은 **완전 additive** — 브라우저가 없으면 결과가 **오늘보다 나빠지지 않습니다.**

### ★ 병렬 조각이 있으면 — 워크트리를 N개 (opt-in)

조각을 **둘 이상** 넘기고 두 축(교집합 0 · 무인 적합도)을 통과하면, 조각마다 워크트리 + 3파일 + **서로 다른 완료문구**를 만들고 명령을 N개 출력합니다. **루프는 사람이 슬라이스마다 세션을 열어 켭니다.** 동시 2~3개 권장.

- **척추(그리고 스캐폴드)는 분기 *이전* 베이스 브랜치에서 1회.** 워크트리마다 스캐폴드를 클론하면 각자 `package.json`·lockfile 을 만들어 **반드시 충돌**합니다.
- **`PROMPT.md` 에 산출물 경로를 슬라이스 폴더로 박으세요** (Step 6 의 기본값은 루트입니다).
- 같은 폴더에서 루프를 두 번 켜면 **뒤가 앞 상태파일을 덮어써 하이재킹**합니다 — 폴더를 갈라야 안전합니다.

절차·실측 근거: [`references/parallel-slices.md`](references/parallel-slices.md)

### ★ 스캐폴드 적용 (Step 4 에서 채택했으면, 격리 안에서)

**절차의 정본은 [`references/scaffold.md`](references/scaffold.md) 입니다** — 클론(임시 폴더로 받아 내용만 얹기) → `verify_bootstrap` → `CLAUDE.partial.md` 병합. **여기에 절차를 다시 쓰지 마세요** (두 벌이 되면 갈라집니다 — 실제로 갈라졌었습니다).

- **스캐폴드 파일 경로는 `${CLAUDE_PLUGIN_ROOT}/scaffolds/<이름>/` 입니다.** `cwd` 아래가 아닙니다 — 설치된 플러그인은 캐시로 복사되므로(위 Step 0 경고) 이 변수를 안 쓰면 **파일을 못 찾아 스캐폴드가 아예 안 됩니다.**

### 먼저 — 완료 문구를 **만드세요. 묻지 말고**

루프의 유일한 정상 출구입니다. **슬러그에서 기계적으로 뽑되 — 반드시 ASCII 로:**

```bash
SLUG=vector-search
BASE=$(echo "$SLUG" | tr 'a-z-' 'A-Z_' | tr -cd 'A-Z0-9')       # 알맹이가 남나?
if [ -z "$BASE" ]; then
  PROMISE=RALPH_HANDOFF_DONE                                     # 한글 슬러그 → 영문 대체
else
  PROMISE=$(echo "$SLUG" | tr 'a-z-' 'A-Z_' | tr -cd 'A-Z0-9_')_DONE
fi
echo "$PROMISE"                                                  # → VECTOR_SEARCH_DONE
```

**길이로 판정하지 마세요.** `목-화면-뼈대` 는 `___DONE`(7글자) 이 돼서 길이 검사를 통과해버립니다. **알맹이가 남았는지**를 보세요 — 위처럼 `_DONE` 을 붙이기 **전에** 검사합니다.

**★ ASCII 여야 합니다.** 완료 문구는 **명령줄 인자 → JSON transcript → perl 정규식 → bash 문자열 비교**를 거칩니다. 한글이 끼면 어디서 깨질지 모르고, 깨지면 **루프가 영원히 안 끝납니다.**

- 슬러그가 한글이면 (`목-화면-뼈대`) → 위 `tr -cd` 가 다 날려서 `_DONE` 만 남습니다. 그때 `RALPH_HANDOFF_DONE` 같은 **영문 대체값**을 쓰세요.
- **`<완료문구>` 를 그대로 두지 마세요.** 실제 문자열을 박으세요 — `PROMPT.md` 안과 실행 명령 **양쪽에 같은 값**으로.
- 훅이 `<promise>…</promise>` **안의 텍스트를 정확히 문자열 비교**합니다. **한 글자만 달라도 루프가 안 끝납니다.**

### ★★ `PROMPT.md` 에 **태그째** 박으세요. 문구만 쓰면 안 됩니다 ★★

```markdown
전부 참이면 — 오직 그때만 — 마지막 메시지에 **아래를 태그째 그대로** 출력하라:

​```
<promise>VECTOR_SEARCH_DONE</promise>
​```

**`<promise>` 태그를 반드시 붙여라.** 훅이 태그 안의 텍스트만 뽑아 비교한다.
태그 없이 문구만 쓰면 — 앞뒤에 한 마디라도 덧붙이는 순간 매치가 깨져
**루프가 영원히 안 끝난다.** 태그로 감싸면 앞뒤에 설명을 붙여도 안전하다.
```

**왜 태그가 필수인지 — 훅 로직을 재현해서 확인한 결과입니다:**

| 루프의 마지막 메시지 | 결과 |
|---|---|
| 문구만 딱, 다른 말 0 (태그 X) | ✅ 종료 |
| `"모든 검증 통과했습니다."` + 문구 (태그 X) | **❌ 루프 계속** |
| 문구 + `"VERIFY.md 확인해주세요"` (태그 X) | **❌ 루프 계속** |
| `<promise>문구</promise>` + 앞뒤 설명 | ✅ 종료 |

**LLM 은 거의 항상 앞에 한 줄 붙입니다.** 태그가 없으면 30회를 다 돌고 `max-iterations` 로 죽어요.

> 이유: 훅의 `perl -0777 -pe 's/.*?<promise>(.*?)<\/promise>.*/$1/s'` 는 **매치가 없으면 원본을 그대로 둡니다.** 그래서 태그가 없으면 `PROMISE_TEXT` 가 **메시지 전체**가 되고, 완료 문구와 안 맞습니다.

`PROMPT.md` 를 만듭니다. **6-1 ~ 6-7 은 전부 실제 무인 실행에서 터진 것들입니다. 하나도 빼지 마세요.**

> **6-8(브라우저 검증)은 성격이 다릅니다** — 무인 실행 사고 기록이 아니라 **신규 기능**이고, 근거는 개발 중 스모크 실측입니다(브라우저 launch 실패). 흉터와 같은 무게로 읽지 마세요.

### 6-1. 상태 판정 — 매 반복 첫 동작

루프는 매 반복 **기억이 없습니다.** 진척을 파일에서 읽게 하세요.

```
② 없음 / 변경이력 0        → ② 를 쓴다 → 커밋 → 멈춘다
② 완성 · ③ 없음 / 0        → ③ 을 쓴다 → 커밋 → 멈춘다

전부 체크 · VERIFY.md 없음  → 검증 돌리고 VERIFY.md
```

**변경이력 엔트리 수가 완성 신호입니다.** `auto-tech-design` 은 **커밋을 안 해서** git 이력으로는 진척을 못 읽습니다.

`auto-tech-design` 은 끝나면 `auto-writing-plans` 를 **자동 호출**합니다 (Step 7). **그걸 하지 말라고 명시하세요.** 안 그러면 한 반복에 다 하려다 컨텍스트가 터지고, 반쯤 쓰다 죽으면 다음 반복이 그걸 완성본으로 오해합니다.

### 6-3. ★ 완료 조건에 사람이 센 숫자를 넣지 마라

```
❌  pytest 25건 통과
✅  pytest 전부 통과 (실패 0 · 에러 0)

**실제로 터졌습니다** — 계획서에 `22건` 이라 적혔는데 본문이 정의한 건 `25건` 이었습니다. **사람이 안 세면 틀릴 수가 없습니다.**

### 6-4. ★ ③.5 검문 관문

**루프가 쓴 계획서로 루프가 채점하면, 오류가 사라지는 게 아니라 안 보이게 됩니다.**

`Task` 도구로 서브에이전트를 **구현자가 아닌 검문자**로 띄우게 하세요:

> 이 계획서만 읽어라. **코드는 절대 보지 마라.**
> 네 목적은 구현이 아니라 **어긋남 찾기**다.
> 1. 완료 조건의 **모든 숫자**를 본문과 대조해라
> 2. 본문에 없는 파일명·함수명이 완료 조건에 있는지 봐라
> 3. 계획서가 정의한 테스트를 **직접 세어** 비교해라
> 4. 요구사항의 FR·NFR·AC 중 **어느 task 에도 안 걸린 게** 있는지 봐라
> 어긋남을 전부 나열해라. 없으면 `어긋남 없음` 한 줄만.

결과를 `REVIEW.md` 에 그대로 적고, 어긋남이 나오면 **계획서(정본)를 고친 뒤** 진행. **코드에서 우회 금지.**

**다른 프롬프트 · 다른 컨텍스트 · 다른 목적** — 이 셋이 달라야 가로질러 읽힙니다.

**막으려 하지 말고 증거를 남기게 하세요:**

### 6-8. ★ 브라우저 검증 — `[e2e]` AC 를 실제로 클릭한다 (있을 때만)

**README 가 한계로 적어둔 *"루프가 렌더를 검증 못 한다"* 를 뒤집는 자리입니다.** 단 **완전 additive** — 브라우저를 못 띄우면 오늘 그대로(BLOCKED)로 떨어지고, 띄우면 한 단계 위로 올라갑니다. **판정은 루프가 런타임에** 합니다.

**`PROMPT.md` 에 이 순서를 박으세요** (자세한 조각·AC→테스트 번역 규칙은 [`references/web-verify.md`](references/web-verify.md)):

1. **능력 감지** — `npx playwright --version` 이 돌고, **브라우저가 실제로 launch 되나** (`--version` OK 여도 시스템 libs 부재로 못 뜬다 — 실제로 터짐. 한 장 찍어 확인), `CLAUDE.md` 에 dev 서버 명령이 있나?
   - 되면 → 2 로. **안 되면** → `[e2e]` AC 를 `BLOCKED.md` 로 (오늘과 동일, 확인 절차째. 최소 호스트면 `sudo npx playwright install-deps` 를 사람 몫으로). **전체 루프는 멈추지 마라.**
2. **각 `[e2e]` AC 는 `e2e/` 아래 Playwright 테스트로 존재해야 한다.** 없으면 그 AC 는 미완료다.
3. **검증 명령**: `npx playwright test --reporter=line` 의 **터미널 출력 전문**을 `VERIFY.md` 에 붙인다. 실패 시 `test-results/` 의 스크린샷 경로도. **스크린샷은 증거다 — 지우지 마라.**
4. **flaky** (retries 후에도 불안정) 면 그 AC 를 무한 재시도하지 말고 `BLOCKED.md` 로 강등해라. `retries: 1` 고정.
5. **결정론적으로 계속 실패**(기능이 그냥 안 됨)하면 6-6 의 *자가 복구 3회 실패 → `BLOCKED.md`* 규칙으로 내린다. 무한 재시도 금지.

**★ 완료 판정 규칙 — 루프가 브라우저 없이 영원히 안 끝나는 걸 막는 핵심:**

> `[e2e]` AC 는 **(Playwright 테스트 통과)** 이거나 **(`BLOCKED.md` 에 기록됨)** 이면 완료로 친다.
> **브라우저를 못 띄우는 걸 완료 조건에 걸어 루프를 매달지 마라.** — 6-3 의 "사람이 센 숫자 금지" 와 같은 계열: 게이트는 기계가 판정 가능한 것에만 건다.

**왜 MCP 가 아니라 CLI 인가** — 무인 루프는 컨텍스트가 생명입니다. Playwright 를 MCP 서버로 붙이지 말고 `npx playwright` **CLI 를 bash 로** 직접 돌리세요 (`playwright-mcp` 자체가 코딩 에이전트엔 CLI 를 권합니다). 서버를 띄워 물고 있으면 매 반복 컨텍스트만 먹습니다.

## Step 7 — 실행 명령 출력 (직접 실행 X)

```
✅ 준비됐습니다.

  워크트리   ../<repo>-ralph-<slug>
  브랜치     ralph/<slug>
  넘긴 것    CLAUDE.md (스택) · <slug>-requirements.md (FR N개) · PROMPT.md
  원본       그대로입니다 (현재 브랜치 무변경)

이제 새 세션에서 이 두 줄을 치세요 — 루프는 실행한 세션을 그대로 잡습니다:

  cd ../<repo>-ralph-<slug> && claude

  /ralph-loop:ralph-loop "PROMPT.md 를 읽고 그대로 따라라. 매 반복 그 파일이 지시서다." --completion-promise "<완료문구>" --max-iterations 30

⚠️ 명령 이름이 /ralph-loop 가 아니라 /ralph-loop:ralph-loop 입니다
   (플러그인명:명령명). 한 줄로 치세요 — 줄바꿈하면 인자가 깨집니다.

⚠️ --max-iterations 를 꼭 주세요. 완료 문구는 하나만 매칭되므로
   반복 상한이 주 안전장치입니다.

아침에 볼 것:
  cat VERIFY.md     ← 검증을 진짜 돌렸나 (없으면 안 돌린 것)
  cat REVIEW.md     ← 검문자가 뭘 잡았나
  cat BLOCKED.md    ← 막힌 것 · 사람이 볼 것
  git log --oneline
```

**"요구사항만" 모드였다면** 병합 전에 개발방향을 꼭 읽어보라고 안내하세요. 기술 결정을 루프가 골랐습니다.

```bash
git diff main..ralph/<slug> -- 'docs/features/*/*-tech-design.md'
git worktree remove ../<repo>-ralph-<slug>     # 버릴 때
```

## 뭘 잃는지 — 시작 전에 알려주세요

| 잃는 것 | 뜻 |
|---|---|
| **설계의 기술 선택권** | `auto-tech-design` 은 *"alternatives 비교 → **recommendation 자동 선택**"* 입니다. 비교는 하지만 **고르는 것도 자기가** 합니다 |
| **계획을 검토할 사람** | 루프가 쓴 계획을 루프가 실행합니다. **자기가 쓴 걸 의심하지 않아요** |

## 이 방식이 안 맞는 경우

- **기술 선택이 중요** → Step 1 에서 "코드만" 을 고르세요
- **요구사항이 모호** → 루프가 추측으로 메웁니다. 요구사항부터 다시
- **되돌리기 어려운 작업 포함** (DB 마이그레이션 · 배포) → 루프는 파괴적 작업이 금지라 전부 `BLOCKED.md` 로 갑니다. 사람이 하세요

## Anti-Patterns

| 안티 패턴 | 이유 |
|---|---|
| `CLAUDE.md` 없이 넘김 | **루프가 스택을 지어낸다.** 이 스킬의 존재 이유 |
| 질문 계획을 예고 안 하고 바로 물음 | **이 스킬의 최대 실패.** 사용자는 질문 수보다 **언제 끝나는지 모르는 것**에 지친다 |
| 예고한 것보다 많이 물음 | 예고가 거짓말이 된다. 예고한 만큼만 |
| `brainstorming` 에 위임 (명시 요청 없이) | 게이트가 5개 더 생긴다. 직접 써라 |
| 워크트리 만들기 전에 승인 요청 | 원본 무변경 + 한 줄로 되돌아감. 파괴적이지 않다 |
| **빈 프로젝트에 워크트리** | **지킬 게 없는데 폴더만 늘린다.** 브랜치만 파라 |
| 기획서가 커밋되는지 **확인 안 하고** 진행 | 회사 자료가 공개 저장소로 간다. **실제로 두 번 터졌다** |
| `<완료문구>` 를 placeholder 로 남김 | 아무도 안 정하면 루프가 안 끝난다. **실제 문자열을 박아라** |
| 완료 문구에 한글 | 명령줄→JSON→perl→bash 를 거친다. **ASCII 강제** |
| 조각을 무인 적합도로만 매김 | **눈으로 볼 수 있나** 도 같이 봐라. 둘은 자주 반대다 |
| 볼 게 없는 조각인데 그냥 넘김 | **실제로 터졌다.** 22개 테스트는 통과했는데 사람이 볼 게 0. 묻지 말고 **초안에 넣어라** |
| **조각을 층으로 자름** (엔진 층 / 화면 층) | **실제로 터졌다(Wayfare S2).** 로직 조각은 정의상 볼 게 없다. **기능으로 수직으로 잘라라** — 입력→로직→화면이 한 조각 |
| "화면은 다음 조각이 담당" 이라며 확인용 화면 뺌 | **면제 사유가 아니다.** 오늘 밤 결과물이 안 보인다는 뜻일 뿐. 로직만 넘길수록 확인용 화면이 **더** 필요하다 |
| "확인용 화면 넣을까요?" 라고 물음 | 멈춤이 3번이 된다. **넣어두고 승인 게이트에서 빼게** 해라 |
| `§5 범위 밖` 에 "프런트" 를 통째로 씀 | **확인용 화면까지 잘려나간다.** "제품용 화면" 이라고 좁혀라 |
| `§2` 에 "화면 없음" 이라고 규정 | 그 뒤로 화면이 자연히 잘린다 |
| 확인용 화면에 디자인 범위를 안 자름 | 루프가 몇 시간을 CSS 에 쓴다 |
| 렌더 확인을 완료 조건에 넣음 | 브라우저가 없어서 루프가 판정 못 한다. `BLOCKED.md` 로 |
| 현재 브랜치에서 산출물 삭제 | 원본이 사라진다. **워크트리 안에서만** |
| 루프를 직접 시작 | 사람이 겁니다. 명령을 **출력만** |
| 완료 조건 숫자를 사람이 셈 | 22 vs 25 회귀 |
| ③.5 검문 생략 | 자기 답안지로 자기 채점 |
| 미승인 요구사항으로 넘김 | 잠기는 게 사람 의도가 아니라 AI 추측이 된다 |
| AC 에 `[unit]`/`[e2e]` 태그 없이 넘김 | 어느 검증기가 판정할지 모호해진다. 태그가 검증 배선의 기준이다 |
| 브라우저가 있는데 `[e2e]` 를 눈으로만 넘김 | **렌더가 기계 검증 안 된다.** 있으면 Playwright 로 클릭해라 |
| 브라우저 못 띄우는 걸 완료 조건에 걸음 | **루프가 영원히 안 끝난다.** `[e2e]` 는 통과 OR BLOCKED 면 완료 |
| flaky E2E 를 무한 재시도 | 루프가 헛돈다. `retries:1` + flaky 면 `BLOCKED.md` 로 강등 |
| Playwright 를 MCP 서버로 붙임 | 매 반복 컨텍스트만 먹는다. `npx playwright` CLI 를 bash 로 |
| 능력 감지를 `--version` 으로만 함 | 브라우저는 못 뜨는데 OK 로 오판 → 런타임 실패. 실제 launch 를 찍어봐라 |
| 기획서 없는데 인터뷰 없이 한 줄로 요구사항 지어냄 | 슬롯이 비어 루프가 추측으로 메운다. 스키마 인터뷰로 채워라 |
| no-doc 인터뷰를 예고·예산 없이 함 | Step 0 예고가 거짓말이 된다. 예고·예산·기본값 |
| `BLOCKED.md` 를 자유 서술로 씀 | 아침 트리아지가 안 된다. 4열 구조화 |
| `verify_bootstrap` green 확인 없이 스캐폴드 위에 루프 | 깨진 베이스라인 → 원인 못 찾는다 |
| 웹 SaaS 아닌데 스캐폴드 강요 | 한 스택은 한 시장만. 안 맞으면 스캐폴드 없이 |
| 빈 프로젝트에서 병렬 루프 시작 | 각자 `package.json`·lockfile 을 만들어 **반드시 충돌.** 공유 척추 먼저 |
| 병렬인데 `REVIEW/VERIFY/BLOCKED` 를 루트에 | 루프끼리 덮어쓴다. 슬라이스 폴더로 내려라 |
| 파일 교집합 있는 조각을 병렬 | 머지 지옥. 의심스러우면 순차 — 아낀 1시간보다 충돌 1건이 비싸다 |
| 병렬 조각들이 같은 완료문구 | 훅이 정확 일치로 본다. 슬러그마다 다른 ASCII 문구 |
