# 큰 앱 슬라이싱 — 척추 하나 + 화면 조각 N개

> SKILL.md "규모의 현실" 의 상세본. **AC 가 수십 개인 다화면 앱**일 때만 쓴다. 한 조각(한 화면)짜리면 이 파일은 아무 일도 하지 않는다.
>
> **주의 — 이건 설계 레시피지 완주 실측이 아니다.** academy-saas 벤치마크는 **통째로** 무인 실행됐고 랄프는 설계(개발방향·구현계획서)까지만 완주, 코드는 사람이 손으로 냈다(37/37). **화면 조각을 무인 루프로 완주시킨 관측은 아직 0건이다.** 아래는 그 벤치마크의 *구조*에서 도출한 방법이며, 표본 1건이다.

## 왜 — 벤치마크가 병목과 (조각내면 될) 구조를 같이 보여줬다

**관측 (academy-saas · 2026-07-23 · 화면 10개 · AC 37개):**

- 무인 루프에 **통째로** 넘겼더니: 랄프가 개발방향·구현계획서·검문까지 **3반복에 완주**했지만 **코드는 완주 못 했다.** 한 반복 = 한 단계라 1스텝/반복이 처리량의 벽. AC 37개 × (테스트+구현) ≈ 50+스텝은 한 번의 무인 실행에 안 끝난다.
- **병목은 실력이 아니라 처리량이다.** 랄프의 설계 품질은 높았다(백트래킹·탐색상한·결정성·독립 검사기까지 스스로 잡음).

**조각내는 게 되려면 — 화면끼리 코드로 안 엮여야 한다. 그런데 그냥 두면 엮인다.**

academy 실물이 그 반례다:

```
Dashboard.tsx  →  import { 배치한다 } from '../timetable/solver'     ← S7 이 S4 폴더를 import (코드 의존!)
전체 10화면    →  import … from '../ui'  (won·강사이름·화면Props…)   ← 공유 헬퍼가 slice 폴더 안에 있음
```

**이대로면 조각이 독립이 아니다** — S7(대시보드)은 S4(시간표)의 `solver` 없이 green 이 안 나고, 아무 조각도 `ui.tsx` 없이 안 돈다. academy 는 **공유물을 slice 폴더(`screens/`)에 잘못 뒀다.** 이게 슬라이싱을 깨는 실수다.

**규칙: 둘 이상의 화면이 쓰는 건 전부 척추가 흡수한다.** 그러면 각 화면 조각은 **척추(+lib)만 import 하고 다른 조각을 import 하지 않아** 독립이 된다.

---

## 레시피

### 1. 척추를 **분기 전에** 세운다 — 공유물을 여기서 다 빨아들인다

척추 = **모든 화면이 공유하고, 새벽 3시에 틀리면 밤이 통째로 날아가는 것.** 창의성 0, 실패 비용 최대. 조각에 맡기지 않는다.

| 척추에 들어가는 것 | 왜 |
|---|---|
| **도메인 타입** (`domain/types.ts`) | N개 화면이 공유. 조각마다 늘리면 같은 개념이 두 이름으로 갈라진다 |
| **저장소 경계** (`store/`) | 화면은 저장 위치를 모른다 |
| **공유 UI 헬퍼** (`lib/ui.ts` — `won`·`강사이름`·`화면Props` 등) | **★ academy 는 이걸 `screens/ui.tsx` 에 뒀다 = 실수.** 전 화면이 쓰니 척추 감이다 |
| **둘 이상이 쓰는 로직** (예: 배치 `solver`·집계 함수) | **★ academy 는 `solver` 를 `screens/timetable/` 에 뒀고 대시보드가 거길 import 했다 = 실수.** 여러 화면이 쓰면 `lib/` 로 올려라 |
| **앱 셸 + 화면 스텁 배선** (아래 ★) | 조각이 `App.tsx` 를 **안 건드리게** — seam 이 미리 다 배선돼 있어야 한다 |
| **디자인 토큰** (`styles.css`) | 조각마다 색·간격을 새로 발명하지 않게 |
| **검증 하네스** (`vite`+`vitest`+`jsdom`·단일파일 빌드) | `pnpm install` 부터 조각이 하면 그 한 번의 실패가 밤을 날린다 |

**★ 셸 스텁 배선 — 이게 "파일 교집합 0" 의 열쇠다.** 척추의 `App.tsx` 가 N개 화면을 **미리 전부 import·분기**하되, 각 화면은 처음엔 "아직 만들지 않았습니다" 를 그리는 **스텁 파일**(`screens/<화면>/<화면>.tsx`)을 가리킨다. 조각은 **자기 스텁 파일 하나만** 채운다 — `App.tsx` 는 건드리지 않는다.

```
// App.tsx (척추, 잠김) — 스텁을 미리 배선해 둔다
import { Students } from './screens/students/Students'   // 처음엔 스텁
switch (화면) { case '학생': return <Students … />; … }   // N개 case 다 있음
```

> academy 는 이걸 안 했다 — `App.tsx` 의 화면자리가 통짜 스텁이었고, 구현하며 import·case 를 **App.tsx 에 직접** 넣었다. 그래서 조각으로 나눴다면 조각마다 공유 `App.tsx` 를 2줄씩 편집해 **교집합이 0 이 아니었을 것.** 스텁 배선이 그걸 막는다.

**척추는 실측 green 을 확인하고 커밋 + 태그(`<repo>-start`) 한 뒤에 분기한다.**

### 2. 화면 목록 → 조각 목록 (한 화면 = 한 조각, 수직)

각 조각은 **그 화면의 AC 서브셋만** 갖고, 입력→로직→화면을 관통(수직). 데이터원 순서로:

```
척추 → students → courses → enroll → timetable → attendance → billing → dashboard/analytics/grades → settings
       앞 화면이 뒤 화면의 데이터를 만든다
```

> **데이터 의존 ≠ 코드 의존.** 수강신청은 학생·강의 **데이터**가 필요하지만, 척추 규칙을 지키면 students **컴포넌트를 import 하지 않는다.** 순서는 "아침에 볼 게 있으려면" 일 뿐 — **뒤 조각도 자기 데모 데이터로 혼자 green 이 난다**(각 조각의 테스트는 자기 데모 데이터를 만들어 쓴다).

### 3. 조각마다 핸드오프 3종 — 완료 문구를 **다르게**

각 조각 = 자기 requirements 서브셋 + 자기 PROMPT + **자기 완료 문구**(`STUDENTS_DONE`·`TIMETABLE_DONE`…, ASCII·유일). requirements 는 그 화면의 AC 만, 점수는 그 조각의 AC 로 센다.

**공유 규칙 (조각이 깨기 쉬운 것):**
- **척추를 다시 세우지 마라** — 설치·타입·저장소·셸·공유헬퍼는 이미 있다. 조각은 **자기 스텁 파일과 그 테스트만** 만든다.
- **다른 조각 폴더를 import 하지 마라** — 필요하면 그건 **척추(domain/lib)** 감이다(위 solver·ui 교훈).
- **`App.tsx` 를 건드리지 마라** — 스텁 배선이 이미 돼 있다.

### 4. 실행 — 순차 기본, 진짜 교집합 0일 때만 병렬

스텁 배선을 제대로 했으면 화면 조각은 **자기 폴더 파일만** 만져 교집합 0 → 병렬 후보. 다만 산출물(`REVIEW/VERIFY/BLOCKED`)·완료문구는 [`parallel-slices.md`](parallel-slices.md) 규칙을 따른다: **순차는 척추 브랜치에서, 병렬은 척추에서 분기한 각자 워크트리/브랜치**에서. **의심스러우면 순차.**

```
세션 1: STUDENTS_DONE    세션 2: COURSES_DONE    …   (조각마다 다른 완료문구)
```

## 점수 — 조각별 + 누적, 경계는 척추가 담당

조각 하나의 점수 = 그 화면 AC 중 green. **화면 조각들이 커버하는 건 기능 AC(academy 기준 AC-1~33)뿐이다.** 경계 AC(academy 의 AC-34~37: 척추 미변경·외부참조 0·껍데기 아님)는 **척추 green 체크가 담당**한다 — 조각이 아니라. 그래서 슬라이스 합의 상한은 33, 경계 4는 척추에서 나와 합이 37이다.

**완주가 아니라 "10화면 중 7개 green" 이 정상적인 밤 결과다.** 통째로 넘겨 0점 받는 것보다, 조각으로 넘겨 부분 점수를 쌓는 게 낫다 — 그게 이 레시피의 전부다.

## academy 매핑 (증거 — AC 는 요구사항 실측, 조각당 개수는 제각각이다)

| 조각 | 화면 | AC | 개수 | 완료 문구 |
|---|---|---|---|---|
| 척추 | domain·store·shell 스텁·공유 UI·공유 로직·토큰·하네스 | AC-34~37(경계) | 4 | (green 확인 후 분기) |
| S1 | students (+상세 프로필) | AC-1~6 | 6 | `STUDENTS_DONE` |
| S2 | courses | AC-7~10 | 4 | `COURSES_DONE` |
| S3 | enroll | AC-11~14 | 4 | `ENROLL_DONE` |
| S4 | timetable (solver·checker) | AC-15~23 | **9** | `TIMETABLE_DONE` |
| S5 | attendance | AC-24~25 | 2 | `ATTENDANCE_DONE` |
| S6 | billing | AC-26~28 | 3 | `BILLING_DONE` |
| S7 | dashboard·analytics·grades | AC-29~31 | 3 | `INSIGHTS_DONE` |
| S8 | settings·데모시드 | AC-32~33 | 2 | `SETTINGS_DONE` |

**조각당 AC 는 2~9개로 제각각이다** — S4(시간표)가 9개로 제일 무겁다(백트래킹·독립 검사기). 통째(33 기능 AC)보다는 작지만 "3~4개" 로 균일하지 않다. 무거운 조각은 더 쪼갤 수도 있다(solver 조각 / 화면 조각). **단, 어떤 화면 조각도 무인 루프로 완주된 관측은 아직 없다** — 크기가 "완주 가능" 이라는 건 통째 실패 + 사람 완성에서 **추론**한 것이지 실측이 아니다.

## 하지 말 것

| 안티 | 왜 |
|---|---|
| AC 수십 개를 통째로 한 PROMPT 에 | 1스텝/반복 × 50스텝 = 무인 실행에 안 끝난다. **academy 관측** |
| 공유물(UI 헬퍼·여러 화면이 쓰는 로직)을 slice 폴더에 둠 | **academy 가 그랬다(ui.tsx·solver → screens/).** 조각 독립성이 깨진다. **척추/lib 로 올려라** |
| 조각이 `App.tsx` 를 편집해 화면을 배선 | 공유 셸을 조각마다 2줄씩 고쳐 교집합이 0 이 아니게 된다. **스텁을 척추에 미리 배선** |
| 조각마다 척추(설치·타입·저장소)를 다시 세움 | 중복 + 갈라짐. 척추는 **분기 전 1회** |
| 데이터 의존을 코드 의존으로 착각 | 뒤 화면은 자기 데모 데이터로 혼자 green. 순서는 "아침에 볼 것" 일 뿐 |
| 조각들이 같은 완료 문구 | 훅이 정확 일치로 본다. 화면마다 다른 ASCII |
| "화면 슬라이스는 밤새 완주된다" 고 단정 | **관측 0건이다.** 통째 실행에서 설계는 완주·코드는 미완주였다. 크기 추론일 뿐 — 실측 뒤 갱신하라 |
| "10화면 다 green 아니면 실패" 라고 봄 | **부분 점수가 정상.** 0점보다 낫다 |
